# Проект: Граф знаний по Москве (Neo4j + AI Agent + FastAPI)

## 1) Цель проекта

Создать граф знаний по Москве с данными о городе (история, дороги, адреса, объекты инфраструктуры, заведения, организации и т.д.) и предоставить два способа доступа:

1. Прямой доступ через `Cypher`-запросы.
2. Доступ через AI-ассистента на естественном языке, который генерирует `Cypher`, выполняет его в Neo4j и возвращает результат пользователю.

---

## 2) Основные требования

- Хранение данных: графовая БД `Neo4j`.
- Инфраструктура: запуск Neo4j в `Docker` с поддержкой `APOC`.
- Данные: сбор из открытых источников, очистка и нормализация перед загрузкой.
- Backend: `FastAPI` с базовыми методами:
  - загрузка данных в БД;
  - выполнение пользовательского Cypher;
  - запрос к AI-агенту.
- Документация: описание архитектуры, модели данных, пайплайнов, API и сценариев запуска.

---

## 3) Архитектура решения

### Компоненты

1. **Data Ingestion Layer**
   - Скрипты загрузки данных из открытых источников.
   - Нормализация и преобразование в формат загрузки для Neo4j.

2. **Graph Storage Layer (Neo4j)**
   - Хранение онтологии (сущностей, связей, атрибутов).
   - Индексы/ограничения целостности.
   - Процедуры APOC для импорта и пост-обработки.

3. **AI Query Layer**
   - LLM-агент, который:
     - принимает вопрос на естественном языке;
     - строит корректный и безопасный Cypher;
     - выполняет его через инструмент доступа к БД;
     - возвращает структурированный ответ.

4. **API Layer (FastAPI)**
   - REST-эндпоинты для данных, Cypher и AI-запросов.
   - Валидация входов, логирование, обработка ошибок.

5. **Ops/Infra Layer**
   - Docker Compose для сервисов.
   - Конфигурация окружений (`.env`).
   - Базовый мониторинг и health-check.

## 4) Онтология графа (этап 1)

### 4.1 Сущности (пример стартового набора)

- `City` (город)
- `District` (район)
- `Street` (улица)
- `Address` (адрес)
- `Road` (дорожная сущность/участок)
- `Place` (объект/место)
- `Organization` (организация)
- `Category` (категория объекта)
- `HistoricalEvent` (историческое событие)
- `Landmark` (достопримечательность)
- `TransportStop` (остановка)
- `MetroStation` (станция метро)

### 4.2 Связи (пример)

- `(:District)-[:PART_OF]->(:City)`
- `(:Street)-[:IN_DISTRICT]->(:District)`
- `(:Address)-[:ON_STREET]->(:Street)`
- `(:Place)-[:LOCATED_AT]->(:Address)`
- `(:Organization)-[:OPERATES_AT]->(:Place)`
- `(:Place)-[:HAS_CATEGORY]->(:Category)`
- `(:Landmark)-[:RELATED_TO]->(:HistoricalEvent)`
- `(:Road)-[:CONNECTS_TO]->(:Road)`
- `(:TransportStop)-[:NEAR]->(:Place)`
- `(:MetroStation)-[:NEAR]->(:Place)`

### 4.3 Атрибуты (пример)

- Общие: `id`, `name`, `source`, `updated_at`.
- Гео: `lat`, `lon`, `geo_hash` (при необходимости).
- Адресные: `house_number`, `postal_code`, `full_address`.
- Исторические: `date_start`, `date_end`, `description`.
- Организации: `inn` (если есть), `website`, `phone`, `work_hours`.

### 4.4 Ограничения и индексы (минимум)

- `UNIQUE` на `id` для ключевых сущностей.
- Индекс на `name` для часто запрашиваемых сущностей.
- Гео-индексируемые поля хранить согласованно (`lat/lon`).

## 5) Сбор и подготовка данных (этап 2)

### 5.1 Источники данных (категории)

- Открытые городские данные.
- Адресные/гео-данные.
- Публичные справочники организаций.
- Исторические источники с открытой лицензией.

### 5.2 ETL/ELT пайплайн

1. **Extract**: выгрузка данных из API/CSV/JSON.
2. **Transform**:
   - нормализация названий и адресов;
   - дедупликация по ключам;
   - маппинг полей на онтологию;
   - валидация обязательных атрибутов.
3. **Load**:
   - генерация импортных файлов (`CSV`) или батч-вставка;
   - загрузка в Neo4j через Cypher/APOC;
   - проверка консистентности после загрузки.

### 5.3 Рекомендуемая структура данных в проекте

- `data/raw/` — исходные файлы.
- `data/processed/` — очищенные и нормализованные наборы.
- `data/mappings/` — правила маппинга.
- `scripts/etl/` — скрипты загрузки/очистки.

## 6) Neo4j + Docker + APOC (этап 3)

### 6.1 Базовые задачи

- Поднять контейнер `neo4j` через `docker-compose`.
- Подключить плагин `APOC`.
- Настроить персистентные тома (`data`, `logs`, `plugins`, `import`).
- Вынести креды и параметры в `.env`.

### 6.2 Минимальные параметры конфигурации

- `NEO4J_AUTH`
- `NEO4J_PLUGINS=["apoc"]`
- `NEO4J_dbms_security_procedures_unrestricted=apoc.*`
- `NEO4J_dbms_security_procedures_allowlist=apoc.*`

### 6.3 Проверка запуска

- Доступ к Neo4j Browser.
- Проверка APOC: `RETURN apoc.version()`.
- Тестовый импорт и тестовый Cypher-запрос.

## 7) AI-агент для генерации Cypher (этап 4)

### 7.1 Функциональные требования

- Принимать вопрос на русском (и опционально на английском).
- Генерировать валидный Cypher по заданной схеме.
- Исполнять только безопасные операции чтения по умолчанию.
- Возвращать:
  - сгенерированный Cypher;
  - результат запроса;
  - краткое текстовое объяснение.

### 7.2 Безопасность и контроль

- Запрет опасных операций (`CREATE`, `MERGE`, `DELETE`, `DROP`) в публичном режиме.
- Лимит результата (`LIMIT`) и таймаут запроса.
- Валидация Cypher перед выполнением.
- Логирование вопроса, Cypher и времени выполнения.

### 7.3 Подход к промптингу агента

- В системный промпт включить:
  - описание онтологии;
  - список допустимых сущностей и связей;
  - правила генерации безопасных read-only запросов;
  - формат ответа JSON.

## 8) Backend на FastAPI (этап 5)

### 8.1 Эндпоинты (MVP)

- `POST /ingest/run`
  - запускает процесс загрузки данных.
- `POST /query/cypher`
  - принимает сырой Cypher, выполняет запрос (с ограничениями/ролями).
- `POST /query/nl`
  - принимает вопрос на естественном языке, вызывает AI-агента, возвращает ответ.
- `GET /health`
  - проверка состояния API и соединения с БД.

### 8.2 Модули backend

- `app/api/` — роуты.
- `app/services/` — логика выполнения запросов и работы с агентом.
- `app/db/` — драйвер Neo4j и утилиты.
- `app/schemas/` — Pydantic-модели запросов/ответов.
- `app/core/` — настройки, логирование, безопасность.

### 8.3 Обработка ошибок

- Ошибки валидации входа.
- Ошибки генерации Cypher.
- Ошибки выполнения в Neo4j.
- Таймауты и пустые результаты.

## 10) Критерии готовности MVP

- Поднят и стабильно работает Neo4j в Docker с APOC.
- Есть импорт минимум одного полноценного набора данных.
- Работают два канала запросов:
  - прямой Cypher;
  - естественный язык через AI-агента.
- FastAPI отдает предсказуемые ответы и корректно обрабатывает ошибки.
- Документация описывает запуск, структуру данных и API.

---

## 11) Риски и меры снижения

- **Низкое качество источников** -> Валидация, дедупликация, доверительные score.
- **Ошибочные Cypher от LLM** -> Строгий prompt + валидация + sandbox read-only.
- **Перегрузка БД** -> Индексы, `LIMIT`, батч-загрузка, таймауты.
- **Сложность онтологии** -> Итеративная версия v1 -> v2, без преждевременного усложнения.
